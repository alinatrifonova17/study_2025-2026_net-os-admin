2.4.1 Установка DNS-сервера



В ходе лабораторной работы я выполнила настройку DNS-сервера.



Сначала я загрузила операционную систему и перешла в рабочий каталог проекта, используя команду:



cd /var/tmp/user\_name/vagrant



Здесь user\_name — это имя пользователя, идентифицирующее меня в системе.



После этого я запустила виртуальную машину server с помощью команды:



make server-up



(или vagrant up server, если работа выполняется в Windows).



После запуска виртуальной машины я вошла в систему под пользователем, созданным в предыдущей лабораторной работе, открыла терминал и перешла в режим суперпользователя:



sudo -i



Далее я установила DNS-сервер BIND и необходимые утилиты с помощью команды:



dnf -y install bind bind-utils



После установки я выполнила тестовый DNS-запрос с помощью утилиты dig:



dig www.yandex.ru



В результате выполнения команды на экран была выведена информация о DNS-запросе, включая:

&#x09;•	тип запроса,

&#x09;•	DNS-сервер, который обрабатывал запрос,

&#x09;•	IP-адрес запрашиваемого домена,

&#x09;•	время ответа сервера.



Я проанализировала выведенную информацию построчно.

2.4.2 Конфигурирование кэширующего DNS-сервера



Анализ конфигурационных файлов



На следующем этапе я изучила содержимое конфигурационных файлов DNS-сервера:



/etc/resolv.conf

/etc/named.conf

/var/named/named.ca

/var/named/named.loopback

/var/named/named.localhost



Я проанализировала назначение каждого файла.

&#x09;•	/etc/resolv.conf — содержит список DNS-серверов, используемых системой.

&#x09;•	/etc/named.conf — основной конфигурационный файл DNS-сервера BIND.

&#x09;•	named.ca — файл с корневыми DNS-серверами.

&#x09;•	named.loopback — конфигурация зоны loopback.

&#x09;•	named.localhost — конфигурация зоны localhost.



⸻



Запуск DNS-сервера



После анализа конфигурационных файлов я запустила DNS-сервер командой:



systemctl start named



Далее я добавила службу DNS в автозагрузку системы:



systemctl enable named





⸻



Проверка работы DNS-сервера



Для проверки работы сервера я выполнила две команды:



dig www.yandex.ru



и



dig @127.0.0.1 www.yandex.ru



Я сравнила полученные результаты.



Разница заключается в том, что:

&#x09;•	первая команда использует DNS-серверы, указанные в системе;

&#x09;•	вторая команда отправляет запрос напрямую локальному DNS-серверу.



⸻



Настройка DNS-сервера по умолчанию



Далее я настроила локальный DNS-сервер в качестве DNS-сервера по умолчанию.



Для этого я изменила настройки сетевого соединения eth0 с помощью NetworkManager:



nmcli connection edit eth0

remove ipv4.dns

set ipv4.ignore-auto-dns yes

set ipv4.dns 127.0.0.1

save

quit



Аналогичные настройки я выполнила для соединения System eth0:



nmcli connection edit System\\ eth0

remove ipv4.dns

set ipv4.ignore-auto-dns yes

set ipv4.dns 127.0.0.1

save

quit



После этого я перезапустила NetworkManager:



systemctl restart NetworkManager



Затем я проверила изменения в файле:



/etc/resolv.conf





⸻



Настройка доступа к DNS-серверу



Для того чтобы DNS-сервер обрабатывал запросы от всех узлов внутренней сети, я внесла изменения в файл:



/etc/named.conf



Я изменила строки:



listen-on port 53 { 127.0.0.1; };



на



listen-on port 53 { 127.0.0.1; any; };



и



allow-query { localhost; };



на



allow-query { localhost; 192.168.0.0/16; };





⸻



Настройка пересылки DNS-запросов



Если маршрутизатор фильтрует DNS-запросы, необходимо настроить пересылку запросов на вышестоящие DNS-серверы.



Для этого в файл named.conf в секцию options я добавила:



forwarders {

10.128.0.240;

80.250.174.240;

};



forward first;



Список DNS-серверов я получила с помощью команды:



cat /etc/resolv.conf



Если вышестоящий DNS-сервер не поддерживает DNSSEC, необходимо добавить:



dnssec-enable no;

dnssec-validation no;

2.4.3 Конфигурирование первичного DNS-сервера



Далее я настроила первичный DNS-сервер.



Сначала я скопировала шаблон файла зон:



cp /etc/named.rfc1912.zones /etc/named/



После этого я переименовала файл:



mv /etc/named/named.rfc1912.zones /etc/named/user.net



Затем я перешла в каталог:



cd /etc/named



После этого я подключила файл зоны в конфигурации DNS-сервера, добавив в конец файла /etc/named.conf строку:



include "/etc/named/user.net";





⸻



Настройка зоны



Далее я отредактировала файл:



/etc/named/user.net



и вместо зоны:



zone "localhost.localdomain" IN {

&#x20;   type master;

&#x20;   file "named.localhost";

&#x20;   allow-update { none; };

};



я указала собственную прямую зону:



zone "user.net" IN {

&#x20;   type master;

&#x20;   file "master/fz/user.net";

&#x20;   allow-update { none; };

};





⸻



Создание каталогов зон



В каталоге /var/named я создала директории для хранения файлов зон:



cd /var/named

mkdir -p master/fz

mkdir -p master/rz





Создание файла зоны



Далее я скопировала шаблон файла:



cp /var/named/named.localhost /var/named/master/fz/



После этого я переименовала файл:



cd /var/named/master/fz

mv named.localhost user.net





Настройка прав доступа



В конце я настроила права доступа для файлов DNS-сервера:



chown -R named:named /var/named



Это необходимо для корректной работы службы named.





\# Ответы на контрольные вопросы



1\. Что такое DNS?

DNS (Domain Name System) — это система доменных имён, предназначенная для преобразования доменных имён сайтов в IP-адреса. Благодаря DNS пользователи могут обращаться к серверам по удобным именам (например, www.example.com), а не по числовым IP-адресам.





2\. Каково назначение кэширующего DNS-сервера?

Кэширующий DNS-сервер используется для хранения ранее полученных DNS-ответов. При повторном запросе того же домена сервер может выдать ответ из кэша, не обращаясь к другим DNS-серверам. Это ускоряет обработку запросов и снижает нагрузку на сеть.





3\. Чем отличается прямая DNS-зона от обратной?

Прямая DNS-зона используется для преобразования доменного имени в IP-адрес.

Обратная DNS-зона используется для преобразования IP-адреса в доменное имя.





4\. В каких каталогах и файлах располагаются настройки DNS-сервера?

Основные настройки DNS-сервера располагаются:

&#x09;•	/etc/named.conf — основной конфигурационный файл сервера BIND.

&#x09;•	/var/named/ — каталог, в котором хранятся файлы DNS-зон.

&#x09;•	/etc/resolv.conf — файл настройки DNS-клиента системы.



Эти файлы отвечают за конфигурацию сервера, описание DNS-зон и настройки использования DNS системой.





5\. Что указывается в файле resolv.conf?

В файле resolv.conf указываются DNS-серверы, которые система использует для разрешения доменных имён, а также дополнительные параметры поиска доменов.





6\. Какие типы записи описания ресурсов есть в DNS и для чего они используются?



Основные типы DNS-записей:

&#x09;•	A — связывает доменное имя с IPv4-адресом.

&#x09;•	AAAA — связывает доменное имя с IPv6-адресом.

&#x09;•	MX — указывает почтовый сервер домена.

&#x09;•	NS — указывает DNS-серверы зоны.

&#x09;•	CNAME — создаёт псевдоним доменного имени.

&#x09;•	PTR — используется в обратных зонах для сопоставления IP-адреса с именем.

&#x09;•	SOA — содержит основную информацию о зоне.







7\. Для чего используется домен in-addr.arpa?

Домен in-addr.arpa используется для обратного DNS-разрешения, то есть для преобразования IP-адреса в доменное имя.







8\. Для чего нужен демон named?

Демон named — это основной процесс DNS-сервера BIND. Он обрабатывает DNS-запросы, управляет зонами DNS и выполняет разрешение доменных имён.





9\. В чём заключаются основные функции slave-сервера и master-сервера?

&#x09;•	Master-сервер хранит основную копию DNS-зоны и отвечает за её изменение.

&#x09;•	Slave-сервер получает копию зоны с master-сервера и используется для резервирования и распределения нагрузки.







10\. Какие параметры отвечают за время обновления зоны?



За обновление зоны отвечают параметры в записи SOA:

&#x09;•	refresh — время проверки обновлений зоны.

&#x09;•	retry — время повторной попытки обновления.

&#x09;•	expire — время действия зоны без обновления.

&#x09;•	minimum — минимальное время хранения записи в кэше.





11\. Как обеспечить защиту зоны от скачивания и просмотра?



Для защиты зоны можно использовать параметр:



allow-transfer



Он ограничивает список серверов, которым разрешено копирование зоны.





12\. Какая запись RR применяется при создании почтовых серверов?



Для почтовых серверов используется запись MX (Mail Exchange).





13\. Как протестировать работу сервера доменных имён?



Работу DNS-сервера можно проверить с помощью команд:

&#x09;•	dig

&#x09;•	nslookup

&#x09;•	host





14\. Как запустить, перезапустить или остановить службу в системе?



Для управления службами используется команда systemctl:

&#x09;•	запуск службы — systemctl start

&#x09;•	перезапуск службы — systemctl restart

&#x09;•	остановка службы — systemctl stop





15\. Как посмотреть отладочную информацию при запуске службы?



Для просмотра логов службы используется команда:



journalctl -u имя\_службы





16\. Где хранится отладочная информация по работе системы и служб? Как её посмотреть?



Отладочная информация хранится:

&#x09;•	в системных журналах systemd

&#x09;•	в каталоге /var/log



Просмотреть её можно с помощью команды journalctl или просмотра файлов в каталоге /var/log.







17\. Как посмотреть, какие файлы использует процесс? Приведите примеры.



Это можно сделать с помощью команд:

&#x09;•	lsof — показывает открытые файлы процессов

&#x09;•	fuser — показывает процессы, использующие файл

&#x09;•	/proc/PID/ — каталог информации о процессе







18\. Приведите примеры изменения сетевого соединения с помощью nmcli.



Примеры команд:

&#x09;•	просмотр соединений

&#x09;•	изменение DNS-адреса

&#x09;•	изменение IP-адреса

&#x09;•	изменение параметров сетевого соединения



Команда nmcli используется для управления сетью через командную строку.





19\. Что такое SELinux?



SELinux (Security-Enhanced Linux) — это система безопасности Linux, которая реализует механизм контроля доступа на основе политик безопасности.





20\. Что такое контекст (метка) SELinux?



Контекст SELinux — это специальная метка безопасности, которая определяет, какие действия процесс может выполнять с файлом или ресурсом системы.





21\. Как восстановить контекст SELinux после изменения файлов?



Контекст можно восстановить с помощью команды:



restorecon





22\. Как создать разрешающие правила SELinux из журналов?



Для этого используется команда:



audit2allow



Она анализирует журнал безопасности и создаёт правила политики.







23\. Что такое булевый переключатель SELinux?



Булевый переключатель SELinux — это параметр политики безопасности, который позволяет включать или отключать определённые правила безопасности.







24\. Как посмотреть список переключателей SELinux и их состояние?



Список можно посмотреть командой:



getsebool -a







25\. Как изменить значение переключателя SELinux?



Для изменения используется команда:



setsebool



Также можно использовать параметр -P, чтобы изменение сохранилось после перезагрузки системы.
